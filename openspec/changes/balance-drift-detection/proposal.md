## Why

F10 (`journal-verification`) just shipped. Every Pending journal now moves to Verified or Failed within ~10 s of being persisted, so the *audit* trail is intact: we know each `journal_entry + ledger_movement` group balances by construction (F06 enforces it before persist, F10 re-checks after). What we still don't know is whether the **cached** `account.balance` column matches the *sum* of an account's movements across all time. Today the only path to a cached balance is through the F06 transfer pipeline (which mutates the cached balance and writes the matching ledger movement inside one transaction, and F06's atomicity is the only thing keeping the two in sync). A single ill-considered manual `UPDATE account SET balance = …` — by an operator, by a backup-restore that ran on a stale ledger, by a future bug in any code that touches `account.balance` — would silently desynchronise the cached balance from the ledger and nothing in the codebase would notice. F10 doesn't help: F10's domain is the journal entry itself, not the cached balance.

F11 is the final core capability slot and a peer to F10. It introduces the continuous background audit that compares each account's cached balance to the sum of its ledger movements, Suspends any account where the two disagree (except the clearing account, which is exempt by spec), and uses a persistent checkpoint over `ledger_movement.id` (already a monotonic `BIGINT IDENTITY` primary key per F02) so the audit never re-processes verified history and never misses new history, even across restarts. With F11 in place, the system has a closed audit loop: F06 keeps writes balanced, F10 verifies the journals are balanced, F11 verifies the cached balances match the journals.

Almost every primitive F11 needs is already in place:

- `ledger_movement.id` is `BIGINT GENERATED BY DEFAULT AS IDENTITY` (F02 / V2 migration) — strictly monotonic, no gaps in committed rows once the transaction commits, perfect cursor.
- `Accounts.findById(AccountId)` (just added by F10) is the suspend-cascade entry point — F11 reuses it.
- `Account.suspend()` is idempotent at the domain layer for already-SUSPENDED, refuses CLOSED → SUSPENDED — F11 reuses the same guarding pattern F10 introduced.
- `BankCoreApplication` already declares `@EnableScheduling`.
- The clearing-account number is already configurable via `bank.clearing-account.number` and read by F08 / F09 — F11 reads the same property to implement the carve-out so there is no risk of name drift.

What F11 has to introduce:

- A persistent **checkpoint** for the cursor. A new tiny single-row table `audit_checkpoint(audit_name VARCHAR(64) PK, last_movement_id BIGINT NOT NULL)` plus a Flyway migration `V4__audit_checkpoint.sql`. One named checkpoint per audit (just `balance_drift` today; F11 is the only consumer).
- A **`LedgerMovements`** port (application module, new) with three methods: `long currentCeiling()` (returns `COALESCE(MAX(id), 0)`), `Set<AccountId> distinctAccountIdsInWindow(long floor, long ceiling)`, and `BigDecimal sumSignedAmountForAccount(AccountId id)` (sums all movements ever, used to compute the canonical balance against which to compare). All three are single aggregate SQL.
- An **`AuditCheckpoints`** port (application module, new) with `long readOrZero(String name)` and `void save(String name, long lastMovementId)`.
- The **`DetectBalanceDrift`** use case (application module, plain Java) that orchestrates one tick.
- A **`BalanceDriftAudit`** transactional facade (infrastructure module, `@Service @Transactional`) wrapping the use case so the spec's "checkpoint advance + suspensions commit together in one transaction" requirement is honoured. F11 is the second place in the codebase where a use case has no controller and so needs a `@Transactional` facade — F08's `OpenAccountService` was the first; we follow that precedent exactly.
- The **`BalanceDriftScheduler`** (`@Component @Scheduled`) and **`BalanceDriftProperties`** (`@ConfigurationProperties`) — same shape as F10's scheduler and properties.

## What Changes

- Introduce the plain-Java `DetectBalanceDrift` use case in `application/src/main/java/com/bank/core/application/audit/`. Constructor takes `LedgerMovements movements`, `Accounts accounts`, `AuditCheckpoints checkpoints`, an `AccountNumber clearingAccountNumber` (the carve-out target), and a constant `String AUDIT_NAME = "balance_drift"`. Single public method `DriftReport audit()` orchestrates one tick:
  1. `long floor = checkpoints.readOrZero(AUDIT_NAME)`.
  2. `long ceiling = movements.currentCeiling()`. If `ceiling <= floor`, return `DriftReport.empty(floor, ceiling)` — the no-op branch.
  3. `Set<AccountId> candidates = movements.distinctAccountIdsInWindow(floor, ceiling)`.
  4. For each candidate `AccountId id` in iteration order:
     - Load `Account account = accounts.findById(id).orElse(null)`. If null, skip (defensive — the FK constraint makes this unreachable).
     - If `account.number().equals(clearingAccountNumber)`, log INFO `clearing-account audit skipped: {} (per balance-drift-detection spec carve-out)` and skip. The carve-out is checked BEFORE the drift comparison so that the clearing account is excluded from the inspected-count of the summary log too — the spec scenario "Clearing account is never auto-suspended" implies the clearing account is not part of the audit's normal flow.
     - If `account.status() != ACTIVE`, skip — the spec forbids re-suspending already-Suspended and never touches Closed.
     - `BigDecimal expected = movements.sumSignedAmountForAccount(id)`; `Money expectedMoney = Money.of(expected.signum() < 0 ? BigDecimal.ZERO : expected)` — the negative-defense is paranoia: the `Money` invariant rejects negatives, but a corrupted ledger sum could be negative; in that case the audit treats the magnitude as zero for comparison purposes and the drift always fires (cached balance is non-negative by `account.balance >= 0` CHECK constraint, expected is "negative-clamped-to-zero", so they disagree).
     - If `expectedMoney.equals(account.balance())`: no drift, `inspected++`, continue.
     - Otherwise: drift detected. `account.suspend()`; `accounts.save(account)`; emit one ERROR log line `balance drift detected on account {} (cached={}, expected={}); account SUSPENDED`. `inspected++`, `drifted++`.
  5. Advance the checkpoint: `checkpoints.save(AUDIT_NAME, ceiling)`. This MUST happen inside the same transactional boundary as the suspensions (enforced by the `@Transactional` on the `BalanceDriftAudit` facade).
  6. Return `DriftReport(floor, ceiling, inspected, drifted)`.

- Introduce the application-module ports in `application/src/main/java/com/bank/core/application/audit/`:
  - `LedgerMovements` interface — three methods: `long currentCeiling()`, `Set<AccountId> distinctAccountIdsInWindow(long floorExclusive, long ceilingInclusive)`, `BigDecimal sumSignedAmountForAccount(AccountId id)`. Javadoc explains why this is a separate port from `JournalEntries`: F11's audit window queries are conceptually about *movements*, not journals (a journal is the F10 unit; a movement is the F11 unit), and bundling them on the existing `JournalEntries` port would awkwardly name a "balance recompute by account" method on a "journal" port. The two adapters can share the same `LedgerMovementRepository`.
  - `AuditCheckpoints` interface — `long readOrZero(String auditName)` returns the persisted last-movement-id for the named audit (or `0L` if no row), `void save(String auditName, long lastMovementId)` upserts the row.
- Add infrastructure adapters in `infrastructure/src/main/java/com/bank/core/infrastructure/persistence/`:
  - `audit/AuditCheckpointEntity.java` — `@Entity @Table("audit_checkpoint")` with `@Id String auditName` and `long lastMovementId`.
  - `audit/AuditCheckpointRepository.java` — `interface AuditCheckpointRepository extends JpaRepository<AuditCheckpointEntity, String>`.
  - `audit/AuditCheckpointsJpaAdapter.java` — `@Component` implementing `AuditCheckpoints`. `readOrZero` returns `repository.findById(name).map(...).orElse(0L)`; `save` is an upsert via `repository.save(new AuditCheckpointEntity(name, lastMovementId))`. The adapter does NOT declare `@Transactional` — it inherits the transactional context from the calling `BalanceDriftAudit` facade so the checkpoint-advance and account suspensions commit together (spec requirement).
  - `ledger/LedgerMovementsJpaAdapter.java` — `@Component` implementing `LedgerMovements`. Backed by a new `LedgerMovementRepository` extending `JpaRepository<LedgerMovementEntity, Long>` (long PK matches `LedgerMovementEntity.id`'s `Long`). The three methods:

Constructor injects `BalanceDriftAudit audit` (the facade, not the use case directly — the facade is where the `@Transactional` lives). `tick()` calls `audit.audit()`, then emits exactly one INFO log line `balance drift tick: floor={}, ceiling={}, inspected={}, drifted={}`. No try/catch — same precedent as F10's scheduler. The two property defaults (30 s cadence, 15 s initial delay) match the spec target of "every 30 seconds" and stagger the audit's first run away from F10's so they don't both fire at second 5.
- Introduce `BalanceDriftProperties` in `infrastructure/src/main/java/com/bank/core/infrastructure/scheduling/`. `@ConfigurationProperties("bank.balance-drift") public record BalanceDriftProperties(long fixedDelayMs, long initialDelayMs)`. The use case does not need a `pageSize` — it processes every distinct account that had a movement in the window, and the spec's "bounded sweep" requirement is the window itself (ticks naturally limit work to "movements committed since last tick" rather than the whole history).

- **Operational notes**:
  - Steady-state behaviour: every 30 s an INFO line `balance drift tick: floor=N, ceiling=N, inspected=0, drifted=0` (assuming no new transfers in the prior 30 s) or `inspected=K, drifted=0` (assuming K touched accounts, all balanced). A `drifted>=1` line is the alert signal.
  - When a drift is detected, the corresponding account is Suspended *immediately* (in the same transaction). All subsequent transfers against that account fail with `AccountInactiveException` (F06 behaviour). The operator playbook: inspect the ERROR log to find the offending account, the cached balance, and the expected balance; investigate the cause (manual DB write? unexpected code path?); decide whether to repair the cached balance or the ledger; manually reactivate the account when satisfied.
  - Clearing-account carve-out: if the clearing-account's cached balance ever drifts, F11 logs INFO but does NOT suspend. The rationale: suspending the clearing account would block every F08 open and F09 seed, halting customer onboarding entirely. The carve-out trades the silent-corruption risk for service availability.
  - Performance: each tick's work is bounded by the number of *distinct accounts touched since the last tick*, not by the size of the ledger. The per-account `sumSignedAmountForAccount(id)` is a single aggregate SQL — H2 / PostgreSQL handle this trivially given F02's `idx_ledger_movement_account_id` index. At 1 transfer/s steady state, an audit tick touches ~60 distinct accounts in a 30 s window, runs 60 aggregate SQLs, and completes well under 100 ms.
