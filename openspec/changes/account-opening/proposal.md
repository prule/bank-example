## Why

F05 just shipped the `account` table, the `Accounts` port, and the `AccountsJpaAdapter`. F06 just shipped the `TransferFunds` use case that moves a positive `Money` amount from one Active account to another inside a single transactional boundary, producing one balanced journal entry. F01 provides the `Account.open(...)` factory; F02 provides the ledger; F07 provides the paired-lock primitive that F06 already consumes. Everything F08 needs is already in place — F08 is a composition change, not a new primitive.

F08 (account opening) is the next slot in the manifest's `[F08, F09]` build group and a hard prerequisite for F09 (dev data seeding). It introduces the **internal** account-opening operation: create a new Active account with the requested number at balance zero, then — if the opening balance is positive — fund it via *exactly one* F06 transfer from the bank's well-known **clearing account**. The whole pipeline runs inside a single transactional boundary so either the new account exists *and* has been funded, or neither effect is observable. There is no HTTP endpoint in this change; opening is invoked internally (by F09 seeding, by future tooling). F09 will land directly on top of this.

This change also makes the manifest's previously implicit clearing-account number explicit and configurable (`bank.clearing-account.number`, default `CLEARING-000`), so F09 seeds the same account F08 funds from with no risk of name drift. The seed of the clearing account itself is F09's responsibility — F08 just *requires* it to exist and fails loudly when it does not.

## What Changes

- Introduce the plain-Java `OpenAccount` use case in `application/src/main/java/com/bank/core/application/account/`. Constructor takes `Accounts`, `TransferFunds`, and an `AccountNumber clearingAccountNumber`. Single public method: `Account open(OpenAccountCommand command)` returning the post-funding aggregate (re-loaded from the port so the returned `balance` reflects the funded value). Orchestration:
  1. Reject the command if an account with the requested number already exists (`accounts.findByNumber(command.number()).isPresent()` → throw `DuplicateAccountNumberException(command.number())`). The DB unique index on `account_number` from F05 is the concurrent-write safety net; this pre-check makes the single-caller path deterministic and produces a domain exception rather than a Spring `DataIntegrityViolationException`.
  2. If the opening balance is positive, confirm the clearing account exists (`accounts.findByNumber(clearingAccountNumber).isEmpty()` → throw `ClearingAccountMissingException(clearingAccountNumber)`). Zero-balance opens skip this step entirely — opening a fresh account at zero must succeed even before the clearing account is seeded (otherwise F09 could not bootstrap).
  3. Create the new aggregate with `Account.open(command.number(), Money.zero())`. The spec explicitly states the new account starts at balance zero *before* any funding step — the funding step is the F06 transfer, not the opening factory call, so that every cent ever credited to the account has a corresponding ledger movement.
  4. Persist the new aggregate via `accounts.save(newAccount)`.
  5. If the opening balance is positive, call `transferFunds.transfer(new TransferCommand(clearingAccountNumber, command.number(), command.openingBalance()))`. F06 handles the source-active check on the clearing account, the debit/credit/journal mechanics, and the F07 paired-lock acquisition.
  6. Reload the new account via `accounts.findByNumber(command.number())` and return it. (After the F06 transfer, the in-memory `newAccount` reference is stale by one credit; the reload is the simplest way to return the post-funding state without coupling F08 to F06's mutation order.)
- Introduce `OpenAccountCommand` as a plain Java record in the same package: `(AccountNumber number, Money openingBalance)`. Compact constructor rejects null fields. A negative opening balance cannot be constructed — `Money` itself rejects negatives at construction (existing F01 invariant), so a negative literal fails before the command record is built. A zero opening balance is explicitly allowed (the spec's "Open with zero balance" scenario).
- Introduce two new domain exceptions in `com.bank.core.domain`, both extending `DomainException`:
  - `DuplicateAccountNumberException(AccountNumber number)` — carrying the offending account number for log/error diagnostics; F08 throws when an open is attempted for a number that already exists.
  - `ClearingAccountMissingException(AccountNumber clearingAccountNumber)` — carrying the configured clearing-account number; F08 throws when a positive-balance open is attempted while the clearing account does not exist. The exception name reads as a system-misconfiguration signal (vs. a generic 404 "not found") so operators see the right alert.
- Introduce a thin transactional facade `OpenAccountService` in `infrastructure/src/main/java/com/bank/core/infrastructure/account/` annotated `@Service @Transactional`. Constructor injects the plain-Java `OpenAccount` use case. Single public method `Account open(OpenAccountCommand command)` delegates straight through. Rationale: F08 is internal (no controller), but the spec's "single logical transactional boundary" requirement still has to land somewhere — and unlike F06 (where the controller is the natural transactional shell), F08 has no HTTP entry. Putting `@Transactional` on a thin infrastructure facade keeps the application module Spring-free (the F00 / `transactional-in-application` convention) and gives downstream consumers (F09 startup runner, future HTTP endpoint) a transactional entry point they cannot forget to wrap. The facade is 5 lines; the test suite calls it directly for the atomicity / rollback scenarios.
- Wire the use case and the facade in `bootstrap/src/main/java/com/bank/core/BankCoreApplication.java`:
  - Add a `@Bean OpenAccount openAccount(Accounts accounts, TransferFunds transferFunds, @Value("${bank.clearing-account.number}") String clearingAccountNumber)` factory method that constructs the plain-Java use case from `AccountNumber.of(clearingAccountNumber)`. Reading the property here keeps the bootstrap layer as the single wiring shell and avoids leaking `@Value` into the application module.
  - The `@Service` annotation on `OpenAccountService` is enough for it to be picked up by component scan; no explicit `@Bean` method is needed.
- Add the property to configuration:
  - `bootstrap/src/main/resources/application.yaml`: add a top-level `bank.clearing-account.number: CLEARING-000` default. The default is intentionally human-recognisable so an operator inspecting the `account` table immediately knows which row is the clearing account.
  - `bootstrap/src/main/resources/application-test.yaml`: declare the same property (no override needed — test code uses the default) to make explicit that the test profile shares the production default and to keep both YAMLs symmetric.
- No HTTP changes. No OpenAPI changes. No new database migration. No edit to F01's `Account`, F02's `JournalEntries`, F05's `Accounts`, F06's `TransferFunds`, or F07's `AccountLocker`. No new Gradle dependency.
- Tests:
  - **Domain unit tests** for the two new exceptions (`DuplicateAccountNumberExceptionTest`, `ClearingAccountMissingExceptionTest`) — accessor round-trip, message contains the offending number, `extends DomainException`.
  - **Application unit test** for `OpenAccountCommand` (`OpenAccountCommandTest`) — null rejection for `number` and `openingBalance`; zero opening balance is accepted; negative opening balance is rejected at the `Money` layer (test the integration to confirm).
  - **Application unit test** for `OpenAccount` (`OpenAccountTest`, JUnit 5 + Mockito):
    - Zero opening balance: creates and saves one Active account at zero, never touches `transferFunds`, never reads the clearing account, returns the saved aggregate.
    - Positive opening balance, clearing account present, target number free: saves the new account, invokes `transferFunds.transfer(...)` with `(clearingAccountNumber, newNumber, amount)` exactly once, returns the reloaded aggregate.
    - Duplicate account number: throws `DuplicateAccountNumberException`, never invokes `accounts.save(...)`, never invokes `transferFunds.transfer(...)`.
    - Missing clearing account (positive open): throws `ClearingAccountMissingException` before any save, never invokes `accounts.save(...)` for the new account.
    - Missing clearing account (zero open): succeeds — clearing-account precondition is funding-only.
    - Argument ordering passed to `TransferCommand` is `(source=clearingAccountNumber, destination=newNumber, amount=openingBalance)` — verified explicitly so a future refactor cannot silently swap source and destination.
  - **Integration test** `bootstrap/src/test/java/com/bank/core/account/OpenAccountServiceIntegrationTest` (`@SpringBootTest`, `@ActiveProfiles("test")`):
    - Pre-seed the clearing account at a known balance via `Accounts.save(...)` in setup.
    - Zero open: `account` row count increases by 1, `journal_entry` row count unchanged, the new account is Active at balance zero, the clearing account is unchanged.
    - Positive open: `account` row count increases by 1, `journal_entry` row count increases by exactly 1, the new account is Active at the requested balance, the clearing account's balance is reduced by exactly the requested amount, the new journal entry's movements are one DEBIT against the clearing-account id and one CREDIT against the new-account id, both for the same amount.
    - Atomic rollback: pre-seed clearing account as `SUSPENDED`. Call `open` with a positive opening balance. Assert the call throws `AccountInactiveException` (from F06's debit of the suspended source), the `account` row count is unchanged (the new account creation rolled back), the `journal_entry` row count is unchanged, and the clearing account's balance / status are unchanged.
    - Duplicate account number: pre-seed an account with number `EXISTS-001`. Call `open(EXISTS-001, ...)`. Assert `DuplicateAccountNumberException` is thrown, no new `account` row, no new `journal_entry` row.
    - Missing clearing account (positive open): do NOT seed the clearing account; call `open(NEW-001, 50.00)`. Assert `ClearingAccountMissingException`, no new `account` row, no new `journal_entry` row.
    - Missing clearing account (zero open): do NOT seed the clearing account; call `open(NEW-002, 0.00)`. Assert the call succeeds, the new account is Active at zero, no journal entry.
  - **ArchUnit / boundary verification**: F00's `applicationHasNoFrameworkDependencies` rule must still pass (`OpenAccount`, `OpenAccountCommand`, both new exceptions are framework-free); `OpenAccountService` resides under `com.bank.core.infrastructure..` so the entity / web confinement rules pass unchanged.

## Capabilities

### New Capabilities
- `account-opening`: Internal use case for creating a new customer account and, optionally, funding it from the bank's well-known clearing account through the F06 transfer pipeline so every credit has a matching ledger movement. Ships the `OpenAccount` use case (plain Java, application module), the `OpenAccountCommand` record, the two domain exceptions (`DuplicateAccountNumberException`, `ClearingAccountMissingException`), the `OpenAccountService` transactional facade (infrastructure module, single `@Transactional` boundary across create + fund), and the configurable clearing-account-number property. Enforces: opening at balance zero never creates a journal entry; opening at positive balance always creates exactly one balanced journal entry via F06; duplicate account numbers are rejected before any write; a missing clearing account causes a positive open to fail loudly with no partial state; the entire operation either commits or rolls back atomically. Negative opening balances are rejected at the `Money` boundary per [[account-domain]].

### Modified Capabilities
None. F08 composes existing capabilities without changing their spec-level contracts:
- F01 (`account-domain`): consumed unchanged. `Money` already rejects negatives; `Account.open(...)` produces a zero-balance Active aggregate; `Account.credit(...)` (invoked transitively via F06) enforces the active-status invariant. No new requirement.
- F02 (`immutable-ledger`): consumed unchanged. The journal entry produced by F06 is the standard PENDING entry; F10 will verify it later. No new requirement.
- F05 (`account-lookup`): the `Accounts` port is consumed via both `findByNumber` (pre-check + clearing-account precondition + post-funding reload) and `save` (new account persistence). Both methods were shipped in F05 precisely for F08/F09 to consume; no spec-level change is needed.
- F06 (`fund-transfer`): consumed unchanged. F08 calls `TransferFunds.transfer(...)` with a freshly constructed `TransferCommand`; F06's atomicity, locking, journal balancing, and exception types all apply transitively. The `TransferFunds.transfer(...)` invocation occurs inside the `OpenAccountService` `@Transactional` boundary, so F06's writes share the same JDBC connection as F08's account-create write and either commit or roll back together. No new requirement in the fund-transfer spec.
- F07 (`transfer-locking`): consumed transitively through F06. F08 does not interact with the locker directly. No new requirement.

## Impact

- **Code**:
  - `domain/src/main/java/com/bank/core/domain/DuplicateAccountNumberException.java` (new).
  - `domain/src/main/java/com/bank/core/domain/ClearingAccountMissingException.java` (new).
  - `application/src/main/java/com/bank/core/application/account/OpenAccount.java` (new).
  - `application/src/main/java/com/bank/core/application/account/OpenAccountCommand.java` (new).
  - `infrastructure/src/main/java/com/bank/core/infrastructure/account/OpenAccountService.java` (new — `@Service @Transactional` facade).
  - `bootstrap/src/main/java/com/bank/core/BankCoreApplication.java` (modified — one new `@Bean OpenAccount` factory method reading `bank.clearing-account.number` and constructing the use case).
- **Configuration**:
  - `bootstrap/src/main/resources/application.yaml` (modified — add `bank.clearing-account.number: CLEARING-000`).
  - `bootstrap/src/main/resources/application-test.yaml` (modified — make the property explicit so the test profile is self-documenting; same default).
- **Schema / migrations**: none. F05's `account` table and F02's `journal_entry` / `ledger_movement` tables already cover everything F08 writes.
- **OpenAPI**: none. F08 has no HTTP surface in this change.
- **Build**: no new Gradle dependencies. The OpenAPI generator runs unchanged. No new Flyway migration.
- **Conventions**:
  - Reaffirms F00's "application is Spring-free": `OpenAccount`, `OpenAccountCommand`, and the two exceptions have no Spring/JPA/openapi imports. ArchUnit `applicationHasNoFrameworkDependencies` continues to pass.
  - Reaffirms F00's "domain is JDK-only": both exceptions extend `DomainException` and import nothing beyond JDK + sibling domain types.
  - Reaffirms F02's `transactional-in-application` precedent: the `@Transactional` annotation lives in infrastructure (`OpenAccountService`), not in the application use case. F06 chose the controller as its transactional shell; F08 has no controller, so a thin `@Service` facade plays the same role.
  - Reaffirms F00's "orchestration-shells-thin": `OpenAccountService.open(...)` is one line delegating to the use case; `BankCoreApplication.openAccount(...)` is one line constructing the use case from injected ports. All decision logic lives in the plain-Java `OpenAccount`.
- **Open decisions**:
  - **Unchanged**: `idempotency` (F06 concern — F08 is internal-only, no caller-supplied idempotency key), `reactivation-playbook` (F11 concern), `scheduler-config-externalised` (F10/F11 concern), `transactional-in-application` (carried forward — F08 honours F02's precedent of putting `@Transactional` in infrastructure).
  - **Closed previously, consumed here without re-litigation**: `self-transfer` (closed by F06 — F08 will never produce a self-transfer because the new account and the clearing account are distinct by construction; the F06 rejection still applies defensively), `lock-wait-timeout` (closed by F07 — transitively in force when F08 invokes F06), `account-status-enum-coverage` (closed by F05 — F08 makes no schema additions), `debit-to-zero` (closed by F01 — applies to F06's debit of the clearing account).
  - **No new open decision opened by this change.**
- **Downstream**:
  - **F09** (dev data seeding) is the immediate consumer. The startup runner will inject `OpenAccountService` and call `open(...)` once per seeded customer account, plus the bootstrap seed of the clearing account itself (a separate `Accounts.save(...)` because the clearing account is opened from nothing — there is no prior funding source). F09's `SEED_DATA` gate ensures the seeder only runs in dev. Idempotency in F09 is "skip if number already exists" before calling `open`, which composes cleanly with the `DuplicateAccountNumberException` semantics here.
  - **F10** (journal verification) sweeps the PENDING journals F08 produces transitively via F06; the spec is unchanged.
  - **F11** (balance drift detection) reconciles the cached balances F08 transitively mutates against the ledger sum; the spec is unchanged. F11 excludes the clearing account from drift suspension (per its own spec) — F08 honours this by funnelling every opening transfer through the clearing account.
  - **Future HTTP account-opening endpoint** (out of scope here) can be a thin `@RestController` that maps a request DTO to `OpenAccountCommand`, calls `OpenAccountService.open(...)`, and returns the created `AccountResponse`. The `@Transactional` is already on the service, so the controller stays a one-liner.
- **Backwards compat**: zero. No prior caller exists for `OpenAccount`; the new property has a sensible default; the new exceptions are reachable only by `OpenAccount` callers. Existing endpoints and tests are untouched.
- **Operational notes**: A startup failure of the `bank.clearing-account.number` property (e.g. operator sets it blank) fails fast at bean creation via `AccountNumber.of(...)`'s null/blank rejection (F01 invariant). A positive open against a missing clearing account fails loudly with `ClearingAccountMissingException`, which an operator alert can match by class name. The atomicity guarantee depends on the single `@Transactional` on `OpenAccountService` — a future refactor that splits the create and fund into two methods on separate beans would break the guarantee and is explicitly called out in the class-level Javadoc.
