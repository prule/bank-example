## 4. JPA entities + Spring Data repositories

- [x] 4.1 `JournalEntryEntity` created. **Deviation**: switched from `@OneToMany(mappedBy = "journalEntry", cascade = CascadeType.PERSIST)` to a unidirectional `@OneToMany @JoinColumn(name = "journal_entry_id") @OrderColumn(...)` with `CascadeType.ALL`. Hibernate cannot combine `mappedBy` + `@OrderColumn` + non-nullable, non-updatable FK column — it tries to INSERT the movement with `journal_entry_id = NULL` and UPDATE later, which violates `NOT NULL`. Unidirectional ownership lets Hibernate include the FK in the INSERT directly.
- [x] 4.2 `LedgerMovementEntity` created. **Deviation**: dropped the `@ManyToOne JournalEntryEntity journalEntry` field in favour of a plain `@Column(name = "journal_entry_id", insertable = false, updatable = false) UUID journalEntryId` field. Necessary for the unidirectional `@OneToMany` ownership pattern (only one side can manage the FK). Queries that previously navigated `m.journalEntry.id` now use `m.journalEntryId` directly.
- [x] 4.3 `JournalEntryRepository` created with `findByStatusOrdered` (paged + ordered) and `sumSignedAmount` queries. The latter updated to reference `m.journalEntryId` instead of `m.journalEntry.id` per the deviation above.

## Implementation notes / deviations from design

Consolidated summary of the deviations recorded inline:

- **`rehydrate(...)` is `public`** rather than package-private. Cross-module access from the mapper requires it. Class Javadoc warns application/use-case code off the path.
- **New `IllegalJournalStatusTransitionException`** instead of reusing F01's `IllegalStatusTransitionException`. Keeps `from`/`to` strongly typed to `VerificationStatus` rather than generalising to `Object`.
- **Unidirectional `@OneToMany @JoinColumn @OrderColumn`** instead of bidirectional `@OneToMany(mappedBy=...)`. Hibernate cannot combine `mappedBy` + `@OrderColumn` + non-nullable, non-updatable FK; the unidirectional form lets the cascade INSERT include the FK directly.
- **`LedgerMovementEntity` exposes `journalEntryId` as a plain `UUID` column** (`insertable = false, updatable = false`) rather than a `@ManyToOne JournalEntryEntity`. Required for the unidirectional ownership pattern. Queries filter on the UUID directly.
- **Adapter takes only `JournalEntryRepository`** in its constructor — no `EntityManager`. Integration tests use `JdbcTemplate` for the raw-insert path. Lighter coupling.
- **Integration tests placed in `bootstrap/src/test/java/`** (not `infrastructure/...`). Infrastructure has no `@SpringBootApplication`; Spring context tests need bootstrap's bootloader.
- **Used `@SpringBootTest` rather than `@DataJpaTest`** so the configured profile-specific H2 (and the `@Component`-scanned adapter) are picked up consistently with the rest of this branch's test approach.
- **Bootstrap `build.gradle.kts` gained three `testImplementation` deps**: `project(":domain")`, `project(":application")`, and `org.springframework.boot:spring-boot-starter-data-jpa`. Required because infrastructure exposes those as `implementation` (not `api`), so they aren't visible on bootstrap's compile classpath transitively. Test-scoped is correct; production code in bootstrap should not depend on domain/application directly.
