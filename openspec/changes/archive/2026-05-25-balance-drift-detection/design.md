## Context

The `balance-drift-detection` capability spec already exists at `openspec/specs/balance-drift-detection/spec.md` with five high-level requirements (periodic-checkpoint-bounded audit, DB-side drift detection, suspend-on-drift with clearing carve-out, concurrent-safe with live transfers, per-tick summary log). F11 is the last manifest slot. Every primitive it needs is on the classpath:

- F02 (`immutable-ledger`) shipped `ledger_movement.id` as `BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` — strictly monotonic, gap-free for committed transactions, perfect cursor.
- F02 also shipped `idx_ledger_movement_account_id`, the index `sumSignedAmountForAccount(id)` will use.
- F05 (`account-lookup`) shipped the `account` table with `balance NUMERIC(19,2) NOT NULL CHECK (balance >= 0)`. The mutable cached column F11 audits.
- F08 (`account-opening`) shipped `bank.clearing-account.number` configurability and the convention that the clearing account is the only legitimate genesis-balance account.
- F10 (`journal-verification`) shipped `Accounts.findById(AccountId)` (which F11 reuses to load candidates by id) and the entire `@Scheduled / @ConfigurationProperties / @Component`-based scheduling shell shape we copy here.
- F08's `OpenAccountService` shipped the `@Service @Transactional` thin-facade pattern that F11 reuses on `BalanceDriftAudit`.

Constraints inherited from F00 and earlier changes:

- **Application module is Spring-free** — `DetectBalanceDrift`, `DriftReport`, `LedgerMovements`, `AuditCheckpoints` have zero Spring/JPA/openapi imports.
- **Domain module is JDK-only** — F11 adds no domain types.
- **Transactional boundaries live in adapters or controllers / facades** (F02 `transactional-in-application`, F08 precedent). F11's transactional boundary lives on `BalanceDriftAudit`, not on the use case and not on the scheduler.
- **One use case, one orchestration shell** (F00 `orchestration-shells-thin`). The scheduler is one delegation + one log line.

Stakeholders:

- The operator who needs balance corruptions caught within ~30 s and contained by an automatic suspend.
- The on-call who interprets the `drifted=K` summary line and follows the manual repair playbook.
- F06 transfer authors who must not have their `@Transactional`-wrapped writes blocked by audit locks.

## Goals / Non-Goals

**Goals:**

- Periodic, bounded background audit that compares cached balance to ledger sum, account by account, for every account that had a movement since the last tick.
- A persistent checkpoint over `ledger_movement.id` so audit progress survives restarts.
- Atomic checkpoint advance + suspensions inside one transactional boundary owned by an infrastructure facade.
- Database-side aggregate queries only — no in-memory ledger iteration.
- Hard carve-out for the clearing account: it is *never* auto-suspended; a clearing-account drift logs INFO but does not change status (the system-availability cost of suspending the clearing account dwarfs the silent-corruption risk).
- Concurrent-safe with live F06 transfers — F11 does not block them, F11 does not miss the movements they commit.
- One INFO log line per tick for liveness; one ERROR log line per drift detection for alerting.

**Non-Goals:**

- Auto-reactivation of drift-suspended accounts. The operator decides when the underlying corruption is repaired; F11 only Suspends.
- HTTP endpoints for the audit. F11 is operator-/auditor-facing through logs, the `audit_checkpoint` row, and `account.status` transitions — not through a public API.
- A general-purpose audit framework. F11 ships one audit (`balance_drift`); the `audit_checkpoint` table is shaped so a second audit could be added with no schema change, but no other audit is in scope here.
- An "ignore this account" allow-list beyond the clearing-account carve-out. If an operator wants F11 to stop alerting on an account, the answer is to fix the cached balance.
- Audit of journal balance (that is F10's job; F11 trusts F10's verdict).
- Pagination *within* a tick. The audit's bound is the per-tick window; the spec does not require an additional inner page limit. At realistic transaction rates the per-tick candidate count is small (≤ a few hundred); revisit only if production rates demand it.

## Decisions

### Decision 1: Cursor over `ledger_movement.id` (BIGINT IDENTITY), not over timestamps

F02 already exposes `ledger_movement.id` as a `BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` — a strictly monotonic, gap-free integer for committed rows. F11's cursor uses this id directly.

**Why id, not timestamp:**

- `entry_timestamp` lives on `journal_entry`, not on `ledger_movement` (movements inherit the journal's timestamp via the FK). Cursoring on a denormalised value would require a JOIN per query.
- `Instant`-typed timestamps have only millisecond resolution; two transfers committed in the same millisecond would share a timestamp and the window `(floor, ceiling]` could either include both or neither based on whether the boundary fell exactly between them. A BIGINT id has no such ambiguity.
- The `id` column is already indexed (it's the primary key); the `WHERE id > :floor AND id <= :ceiling` predicate is a one-segment B-tree range scan.

**Why `(floor, ceiling]` (exclusive floor, inclusive ceiling):**

- Symmetric with the spec's wording ("highest ledger movement id that has been audited" = inclusive upper bound of the previously-processed window).
- A new audit on a fresh DB starts with `floor=0` and gets every committed row (since `id > 0` includes everything).
- A subsequent audit starts with `floor=<previous ceiling>` and excludes the previously-audited rows.

### Decision 2: Capture `ceiling` once at tick start, not per query

The use case calls `movements.currentCeiling()` exactly once at the top of `audit()` and stores the returned value. All subsequent queries in the tick use that captured value as the upper bound.

**Why:**

- The spec requires that movements committing *after* the audit captured `ceiling` are deferred to the next tick — "they are never missed, never double-counted". If the use case re-queried for the current max id during each per-account `sumSignedAmountForAccount(...)` (which sums *all* movements, not the window — see Decision 5), the ceiling would drift inside a single tick and a transfer racing the audit could be partially observed.
- F02's `ledger_movement.id` is a database `IDENTITY` whose value is reserved at insert time but only visible to other transactions after commit. So `MAX(id)` returns the highest *committed* id. A transfer that started before the audit's `MAX(id)` query but committed after will have an id higher than the captured `ceiling` and will be picked up by the next tick — exactly the spec's required behaviour.

**Trade-off accepted:** the per-account sum query (`sumSignedAmountForAccount(id)`) is *not* windowed — it sums every movement for that account across all time. So if a transfer for one of the candidate accounts commits between the `currentCeiling()` call and the per-account sum query, that new movement *will* be included in the sum but its contribution to the cached balance happened *before* the sum query (because the transfer committed both writes atomically). The sum and the cached balance are still consistent. The transfer that committed after `ceiling` is also a candidate for the *next* tick, which will re-sum the account and find it still in balance. So there is no double-counting and no missed drift; the only cost is one redundant re-sum on the next tick. See Decision 5 for why we don't window the per-account sum.

### Decision 3: Separate transactional facade `BalanceDriftAudit` owning the @Transactional

The `@Transactional` annotation lives on `BalanceDriftAudit` (infrastructure module), not on the use case (application module is Spring-free) and not on the scheduler (the spec wants the transaction to wrap the use case's full execution, not the scheduler's logging step).

**Why a facade and not @Transactional on the scheduler:**

- The scheduler's job is "log the result and continue". Putting `@Transactional` on the scheduler would mean a log statement participates in the audit transaction — harmless but semantically wrong. The boundary should be exactly the use case's `audit()` call.
- The scheduler injects `BalanceDriftAudit` (the facade), not `DetectBalanceDrift` (the use case) directly. The application use case is *not* a Spring `@Bean` consumed by callers that need the transactional guarantee — only the facade is. This mirrors F08 exactly (`OpenAccountService` wraps `OpenAccount`; controllers/callers consume the service, not the use case).

**Why the facade is in `infrastructure/.../audit/`, not in `infrastructure/.../scheduling/`:**

- The facade is a *transactional shell* around the application use case — it has nothing to do with scheduling cadence. Sharing a package with the scheduler would imply they're a unit. Keeping them in separate packages (`audit/` for the facade, `scheduling/` for the scheduler) signals "the use case has a transactional boundary; the scheduler happens to be one of its callers, but the boundary is independent of the scheduler".

### Decision 4: Checkpoint advances on every tick, including the no-op path

When `ceiling <= floor` (no new movements since the previous tick) or when the candidate set is empty, the use case still calls `checkpoints.save(AUDIT_NAME, ceiling)`. The save is a no-op-on-data (the value is unchanged) but it exercises the same code path as the non-empty case.

**Why:**

- Simpler control flow: one save call in one place, no branching on "did we do work this tick".
- The save is cheap — a single primary-key UPSERT against a one-row table.
- An always-write checkpoint makes the integration test's "checkpoint advances atomically with suspensions" scenario uniform: every tick the checkpoint row's transactional write is observable, even ticks with `drifted=0`.

**Alternative considered:** skip the save when there's no work to do, to keep `audit_checkpoint`'s `xmin` (or H2 equivalent) clean. Rejected — the simplicity win is real and the row-version churn cost is trivial.

### Decision 5: Per-account sum is across all time, not just the window

`LedgerMovements.sumSignedAmountForAccount(id)` returns the sum of every movement ever for that account, not just movements in `(floor, ceiling]`.

**Why:**

- The cached `account.balance` represents the sum of *every* movement ever. To detect drift, we must compare like-with-like: total cached balance vs total ledger sum.
- A windowed sum would only detect drift that occurred *within the window*. If a corruption happened months ago and no new movements have hit that account since, no audit tick would ever notice — because the account never falls into a window's candidate set. We accept this limitation explicitly: F11 only audits accounts that had a movement in the window, not every account.
- The "audit only accounts in the window" choice is the spec's: "The set of accounts to recheck SHALL be identified by the movements in the window `(floor, ceiling]`." The all-time sum then ensures the comparison is meaningful for the accounts we do audit.

**Trade-off accepted:** an account that drifts but never sees another transfer will not be detected by F11. The operator's quarterly full-audit script (out of scope) can cover that gap. F11 catches drift that happens to *active* accounts within ~30 s of the next movement.

### Decision 6: Clearing-account carve-out checked *before* `inspected++`

The carve-out skip happens before incrementing `inspected`. The summary log's `inspected=K` therefore reflects "non-clearing accounts audited this tick", not "all candidates processed". The clearing account is treated as outside the audit's flow.

**Why:**

- The spec scenario says the clearing account "MAY log the discrepancy, but the status remains unchanged" — it explicitly does NOT say the clearing account participates in the audit's normal accounting. Treating it as a separate code path matches that intent.
- Operationally, the `inspected` count is what an operator uses to estimate audit cost. Including the clearing account would inflate the count by 1 every tick (the clearing account appears in nearly every window's candidate set, since most transfers touch it).

**Alternative considered:** count the clearing account in `inspected` and add a separate `clearingSkipped` counter to `DriftReport`. Rejected — the spec doesn't require it, and the extra counter would clutter the summary log line without adding operator value.

### Decision 7: Two ports, not one

We introduce `LedgerMovements` and `AuditCheckpoints` as separate application-module ports rather than bundling everything on `JournalEntries`.

**Why two ports, not extension of `JournalEntries`:**

- `JournalEntries` is the F02 / F10 port for *journal* lifecycle (save, load by id, page by status, balance check by id). Adding `currentCeiling`, `distinctAccountIdsInWindow`, `sumSignedAmountForAccount` would awkwardly mix journal-shaped methods with movement-shaped methods on the same interface.
- `AuditCheckpoints` is a different aggregate entirely (a single key-value row per audit). Bundling it onto either `JournalEntries` or `LedgerMovements` would tightly couple unrelated concepts.

**Why not one port `Audit` carrying everything F11 needs:**

- Hexagonal architecture pins ports to *aggregates*, not to use cases. F11 happens to need three aggregates' worth of data; that's the spec, not the architecture. A single fat "audit" port would make the next audit (if there ever is one) tempted to copy-paste and grow.

### Decision 8: No per-account error isolation inside the audit

Unlike F10's `VerifyPendingJournals.sweep()` which catches per-journal exceptions and continues, F11's `DetectBalanceDrift.audit()` does NOT wrap per-candidate processing in try/catch. An exception loading or saving any single account aborts the tick.

**Why the asymmetry:**

- F10's per-journal failure mode (e.g. a single corrupted journal that can't be balanced) is a *known* recoverable hazard — the journal is exactly the unit F10 was designed to triage. Continuing to the next journal is the right behaviour.
- F11's per-candidate operations are `accounts.findById(...)` and `accounts.save(...)` — both adapter methods backed by JPA. A failure on either is far more likely to be a transient infrastructure issue (connection drop, deadlock) that will affect every subsequent candidate in the same tick too. Catching and continuing would just produce N redundant ERROR lines and a misleading "audit completed" summary.
- The whole-tick transactional boundary makes per-candidate isolation impossible anyway: catching the exception doesn't roll back the partial work, it just lets the loop finish, at which point the surrounding `@Transactional` would still commit. Either we keep the boundary (and let the exception propagate, rolling everything back) or we drop the boundary (and lose the spec's "atomic checkpoint + suspensions" guarantee). We keep the boundary.

**Trade-off accepted:** a single bad account in a tick fails the whole tick. The next tick re-loads the same candidates (because the checkpoint didn't advance) and tries again. The retry succeeds if the underlying issue was transient. The Spring scheduler's WARN line on the failed tick is the operator's signal.

## Risks / Trade-offs

[Risk] A pathological `bank.balance-drift.fixed-delay-ms` (e.g. set to 1) overwhelms the connection pool with audit transactions and starves F06 transfers. → **Mitigation**: `BalanceDriftProperties` rejects non-positive values via the WARN-on-fallback compact-constructor pattern (same as `JournalVerificationProperties`). Even with `fixedDelayMs=1`, the use case's `currentCeiling()` returns quickly on an idle ledger and the audit completes in milliseconds; only sustained load matters. Observable via the per-tick summary log.

[Risk] The carve-out makes the clearing account a silent corruption vector — if its cached balance drifts, no automated remediation fires. → **Mitigation accepted**: this is the spec's intent. The INFO log line is the only automated signal; an operator alert pipeline (out of F11's scope) should pattern-match on `clearing-account audit skipped` and page on it. Suspending the clearing account would block customer onboarding entirely — the availability cost dwarfs the corruption risk.

[Risk] An account that drifts but never sees another movement is invisible to F11. → **Mitigation accepted**: see Decision 5. F11 is a "drift-aware audit on active accounts" not a "full ledger reconciliation". A separate quarterly job (out of scope) covers idle accounts.

[Risk] The `BalanceDriftAudit` `@Transactional` boundary wraps an entire tick. Under sustained high transfer volume, a large candidate set causes a long-running transaction that holds locks. → **Mitigation**: per Decision 5, the per-account sum query is a read against an indexed column and does not lock rows. The only writes are `account.save(...)` for drifted accounts (`UPDATE account SET status = 'SUSPENDED' WHERE id = ?`) and the single `audit_checkpoint.save(...)`. Lock contention is bounded by the count of drifted accounts (typically zero). For realistic workloads the per-tick transaction is sub-second.

[Risk] F11 and F10 both run on the default Spring `TaskScheduler` thread pool. If one slow tick blocks both, audits stop. → **Mitigation**: Spring Boot's default `@EnableScheduling` pool size is 1, so technically they can interleave but never run truly concurrently. A sufficiently slow tick of either could delay the other. Mitigation: the production cadences (10 s for F10, 30 s for F11) plus the typical tick duration (sub-second) leave ample slack. If load grows, a `@Bean ThreadPoolTaskScheduler` with `poolSize=2` is a one-line change. Documented but not implemented in this change.

[Risk] A future change adds a second audit and reuses `audit_checkpoint`. The shared single-row-per-name table is fine, but the constant `"balance_drift"` is hard-coded inside `DetectBalanceDrift`. → **Mitigation accepted**: the constant is a `public static final String AUDIT_NAME` on `DetectBalanceDrift`, so a future audit (e.g. `DetectIdleAccounts` with name `"idle_accounts"`) can pick its own constant. Test asserts the literal value.

[Risk] The integration test relies on side-channel JDBC writes to fabricate drift. If the schema changes, the test breaks silently. → **Mitigation**: the side-channel writes use the `JdbcTemplate` already wired in the test, and the assertions go through the `Accounts` port (which validates the schema match via `ddl-auto=validate` at context start). A schema change would either fail the context start or fail the read assertion — never silently.

[Risk] When the operator manually reactivates a drift-suspended account (out-of-band SQL), F11 will re-detect the same drift on the next tick that the account appears in a window. → **Mitigation accepted**: the operator's repair playbook MUST fix the cached balance (or the ledger) before reactivating. If they reactivate without repairing, the next tick correctly re-suspends. This is the right behaviour: drift is drift.

## Open Questions

None — all decisions are bounded by existing spec requirements and conventions established by F02 / F08 / F10. No external dependency, no HTTP contract, no third-party integration remains undecided.
