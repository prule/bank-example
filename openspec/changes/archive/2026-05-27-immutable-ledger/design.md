## Context

The Bank Core application requires an immutable double-entry ledger database schema and rich model in the `domain` module, fully decoupled from database adapters, to permanently record every movement of funds. 

This design implements the append-only ledger system, ensuring that journal entries are balanced and immutable once stored, and provides continuous, deterministically ordered pages of entries for background sweepers.

## Goals / Non-Goals

**Goals:**
- Implement immutable domain types in the `domain` module: `JournalEntry`, `Movement` (record), `MovementType`, `VerificationStatus`, and `JournalEntryId`.
- Add Flyway schema migrations to provision the H2 SQL `journal_entry` and `ledger_movement` tables.
- Set up a globally monotonic primary key `id` (`BIGINT IDENTITY`) on the `ledger_movement` table to act as a secure cursor for background verification.
- Implement package-private JPA entity mappings and the plain-Java `JournalEntries` application port.
- Build the `JournalEntriesJpaAdapter` executing a single database-side aggregate query (`isBalanced`) to verify balanced journals.

**Non-Goals:**
- Modifying the existing `Account` balance directly within the ledger persistence layer (the ledger solely documents history; balance snapshots are updated separately).

## Decisions

### 1. Domain Models and Double-Entry Invariants
We enforce all accounting validation at the core domain boundary:
- **`Movement`**: An immutable Java `record` carrying the `AccountId`, `Money` amount, and `MovementType` (DEBIT/CREDIT). Rejects null properties and zero/negative amounts (`InvalidAmountException`).
- **`JournalEntry`**: An immutable entity managing the verification status lifecycle.
  - Implements static factory creation: `public static JournalEntry create(...)` requiring at least two movements.
  - Validates that the sum of CREDITs matches the sum of DEBITs exactly (`creditSum.equals(debitSum)`), throwing a custom `UnbalancedJournalException` otherwise.
  - Verification status transitions from `PENDING` to either `VERIFIED` or `FAILED` via explicit named operations `markVerified()` and `markFailed()`. Any illegal transitions throw `IllegalJournalStatusTransitionException`.

*Rationale*: Prevents the creation of unbalanced transactions or illegal status updates in memory.

### 2. Monotonic Database-Generated Movement Cursor
- The primary key `id` of `ledger_movement` is mapped as a `BIGINT GENERATED BY DEFAULT AS IDENTITY`.
- The domain `Movement` record does not carry this ID; it is only materialized upon persistence mapping in `LedgerMovementEntity`.

*Rationale*: Guarantees a globally ordered sequence of ledger operations across the entire bank core system, enabling sweepers to safely traverse movements using a persistent cursor.

### 3. Package-Private Hibernate Entity Setters
To prevent accidental modification of persisted ledger rows in Java:
- `JournalEntryEntity` and `LedgerMovementEntity` map to database tables.
- All column fields on `LedgerMovementEntity` and non-status fields on `JournalEntryEntity` have package-private modifiers and lack public setters.
- The `JournalEntriesJpaAdapter` saves domain objects by inserting new entities or updating only the `verification_status` column.

*Rationale*: Enforces the append-only constraint on a database/Hibernate mapping level, preventing mutability drift.

### 4. Single-Query Database Reconciler (`isBalanced`)
To check if a journal balances at database-level without loading records into memory:
- `JournalEntriesJpaAdapter` issues a single native aggregate SQL query:
  ```sql
  SELECT COALESCE(SUM(CASE WHEN movement_type = 'CREDIT' THEN amount ELSE -amount END), 0)
  FROM ledger_movement WHERE journal_entry_id = :id
  ```
- The returned `BigDecimal` is checked against `BigDecimal.ZERO` using `signum() == 0`.

*Rationale*: Significantly optimizes performance by offloading summation to the database engine and avoiding JVM memory bloating.

## Risks / Trade-offs

- **[Risk]** Float Serialization rounding: The database numeric field might introduce slight precision drift compared to domain `Money` values.
  - *Mitigation*: Configure Flyway columns with `NUMERIC(20, 2)` and map JPA properties to `BigDecimal` directly. The JPA adapter will reconstitute values using the standard `Money` factory which enforces `2` decimal places with `HALF_UP`.
