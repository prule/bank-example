## Why

F04 shipped the canonical OpenAPI contract with an `ErrorEnvelope` schema whose `code` field is a free-string placeholder. Every endpoint that follows (F05 account lookup, F06 fund transfer, F08 account opening) needs a fixed taxonomy of error codes and a guarantee that errors come out in the same shape, with the same fields, and never leak internals — otherwise each capability invents its own error format under deadline pressure. F03 is named alongside F04 in the manifest's `[F03, F04]` build slot because the envelope is useless without the codes that populate it and the runtime handler that produces them. Locking this contract in before the first business endpoint is the cheapest moment to do it.

## What Changes

- Tighten the `ErrorEnvelope.code` field in [bootstrap/src/main/resources/openapi/schemas/error-envelope.yaml](bootstrap/src/main/resources/openapi/schemas/error-envelope.yaml) from a free `string` to an `enum` of the five canonical values: `INSUFFICIENT_FUNDS`, `ACCOUNT_INACTIVE`, `RESOURCE_NOT_FOUND`, `BAD_REQUEST_PAYLOAD`, `INTERNAL_SERVER_ERROR`. The OpenAPI generator will emit a typed enum nested inside the generated `ErrorEnvelope` DTO; hand-written handlers reference enum constants, not strings.
- Document the canonical code → HTTP status mapping in the OpenAPI document (description + extension), matching the published spec: 400 for `INSUFFICIENT_FUNDS`/`ACCOUNT_INACTIVE`/`BAD_REQUEST_PAYLOAD`, 404 for `RESOURCE_NOT_FOUND`, 500 for `INTERNAL_SERVER_ERROR`.
- Add a `@RestControllerAdvice` global exception handler under `com.bank.core.infrastructure.web.error.GlobalExceptionHandler` that converts framework-thrown exceptions into the canonical envelope: validation failures (`MethodArgumentNotValidException`, `ConstraintViolationException`) → 400 `BAD_REQUEST_PAYLOAD`, body parse failures (`HttpMessageNotReadableException`) → 400 `BAD_REQUEST_PAYLOAD`, missing handlers/static resources (`NoHandlerFoundException`, `NoResourceFoundException`) → 404 `RESOURCE_NOT_FOUND`, and a catch-all `Exception` handler → 500 `INTERNAL_SERVER_ERROR` with a generic message (no stack traces, no class names, no SQL, no package paths leaked to clients).
- Wire `spring.mvc.throw-exception-if-no-handler-found=true` and disable Spring's default static resource fallback so `NoHandlerFoundException` actually fires for unknown paths — and the canonical 404 envelope is what clients see, not Spring's default Whitelabel page.
- Log every catch-all 500 at `ERROR` level with the full stack trace, request method, and request path, so operators see the cause server-side while clients see only a stable code.
- Hooks for the domain exception mappings that F01/F06 will introduce: leave `INSUFFICIENT_FUNDS` and `ACCOUNT_INACTIVE` handler stubs out for now (the exception classes don't exist on this branch yet). The published spec carries these mappings as the *contract* the system must honour once those exceptions land; F01 and F06 are responsible for adding the `@ExceptionHandler` entries when they introduce the exception types. The handler class is structured so adding them is a one-line addition each.
- Add controller-level smoke tests against a tiny throwaway test controller (test-only, no production endpoint) to exercise each handler path end-to-end: validation, parse failure, no-handler, generic exception. The tests assert the response status, the envelope shape, and the absence of any leakage in the `message` field.

No new public business endpoint ships with F03. The deliverable is the envelope schema + code enum + handler + tests, ready for F05/F06/F08 to consume.

## Capabilities

### New Capabilities
- `api-error-contract`: Single stable error envelope (`code`, `message`, `timestamp`) defined once in the OpenAPI contract and referenced by every error response; canonical taxonomy of error codes with fixed HTTP status mappings (`INSUFFICIENT_FUNDS`, `ACCOUNT_INACTIVE`, `RESOURCE_NOT_FOUND`, `BAD_REQUEST_PAYLOAD`, `INTERNAL_SERVER_ERROR`); validation, parse, no-handler, and catch-all exception handling in a single `@RestControllerAdvice` that maps to the envelope without leaking internals.

### Modified Capabilities
- `contract-first-api`: The shared error schema requirement (already in the published spec) gains a concrete code enumeration. The `code` field tightens from a free-string with a "taxonomy defined in F03" note to a proper `enum` with the five canonical values. The "exactly once in `components.schemas.ErrorEnvelope`, `$ref`d everywhere" pattern is unchanged — only the schema's internal definition tightens. A small delta spec records the schema-level shift.

## Impact

- **Code**: Adds [GlobalExceptionHandler.java](infrastructure/src/main/java/com/bank/core/infrastructure/web/error/GlobalExceptionHandler.java) under `com.bank.core.infrastructure.web.error`. Adds (optional) `infrastructure/src/test/java/com/bank/core/infrastructure/web/error/` test-only controller plus per-handler integration tests. Tightens [bootstrap/src/main/resources/openapi/schemas/error-envelope.yaml](bootstrap/src/main/resources/openapi/schemas/error-envelope.yaml) so the generator emits a typed `CodeEnum`.
- **Generated code**: The next `./gradlew :infrastructure:openApiGenerate` regenerates `com.bank.core.dto.ErrorEnvelope` with `CodeEnum` as a public nested enum. The hand-written handler uses `ErrorEnvelope.CodeEnum.INSUFFICIENT_FUNDS` etc. — no parallel Java enum needed; the OpenAPI document is still the single source of truth.
- **Configuration**: Adds `spring.mvc.throw-exception-if-no-handler-found: true` and `spring.web.resources.add-mappings: false` to `bootstrap/src/main/resources/application.yaml` so `NoHandlerFoundException` is reachable.
- **Dependencies**: No new dependencies — Spring Boot's web/validation starters from F00 already supply every exception type and annotation needed. Jackson is on the classpath for JSON serialisation of the envelope.
- **Downstream**: Unblocks F05 (404 `RESOURCE_NOT_FOUND` for missing accounts), F06 (`INSUFFICIENT_FUNDS`/`ACCOUNT_INACTIVE` on failed transfers — they add `@ExceptionHandler` entries to the same class when F01 lands the exception types), F08 (account opening flows through the same envelope), and any future capability that returns a `4xx`/`5xx`.
- **Tests**: New per-handler integration tests under `bootstrap/src/test/java/com/bank/core/web/error/`. F00 ArchUnit boundary tests continue to pass — the handler lives in `infrastructure` (Spring is allowed), references generated DTOs (allowed in infrastructure), and does not touch domain/application.
- **No backwards-compat concern**: nothing on this branch ships an error response today (no real endpoint exists yet), so tightening `code` from free string to enum is a definitional refinement, not a breaking change for any client.
